## 4. qBittorrent + Gluetun kill-switch (nftables)

Goal: **qBittorrent must only ever reach the Internet through the VPN tunnel**. If Gluetun misbehaves or VPN drops, traffic must not fall back to WAN. Design assumes: qB shares `Network=container:gluetun` and Gluetun has `/32` 172.20.20.239. 

### 4.1 Concept

On deimos:

* All traffic from 172.20.20.239 **towards WAN interface (e.g. `enp3s0`)** is blocked **except** to the VPN provider endpoint IP/port.
* If VPN is up: host only sees encrypted UDP to that endpoint → allowed.
* If VPN is down and container tries to route arbitrary traffic to WAN: destination ≠ VPN endpoint → dropped.

### 4.2 nftables config

`/etc/nftables.conf` (add to your existing rules):

```nft
table inet vpn_killswitch {
    set vpn_endpoints {
        type ipv4_addr
        flags constant
        elements = { 203.0.113.10 }    # <-- replace with your VPN provider endpoint IP(s)
    }

    chain output {
        type filter hook output priority 0; policy accept;

        # Kill-switch for Gluetun/qB container IP
        ip saddr 172.20.20.239 oifname "enp3s0" ip daddr != @vpn_endpoints counter drop
    }
}
```

If your provider uses multiple endpoints, just add them to the set.

**Apply + test**

```bash
nft -f /etc/nftables.conf
nft list ruleset
```

* With Gluetun configured but VPN **stopped inside** container, try downloading via qB → traffic should fail; `nft monitor` should show `drop` counters increasing.

Optional: add a dedicated systemd unit:

`/etc/systemd/system/nftables-killswitch.service`:

```ini
[Unit]
Description=NFTables kill-switch for Gluetun/qB
Before=network-online.target
Wants=network-online.target

[Service]
Type=oneshot
ExecStart=/usr/sbin/nft -f /etc/nftables.conf
RemainAfterExit=yes

[Install]
WantedBy=multi-user.target
```

## 5. How to configure WireGuard on client devices

Server (ER605) per design:

* Endpoint: `vpn.abhaile.dedyn.io:51820`
* Server tunnel: `172.20.90.1/24` (VLAN 90) 

### General guidance

* **Use the exact AllowedIPs** model from the spec for each profile (next section).
* **DNS enforcement:**

  * Set `DNS = 172.20.20.235, 172.20.20.236` in the client config.
  * On mobile (Android/iOS), the WG app will push these DNS servers while tunnel is up.
  * On desktop:

    * Linux: allowed via resolvconf or NetworkManager; ensure no other DNS is configured with higher priority.
    * Windows/macOS: WireGuard app handles DNS per-tunnel; ensure “DNS over HTTPS” in browsers is disabled to respect internal split-horizon.
* **“Block outside VPN” (kill-switch):**

  * Android: enable “Block connections without VPN” for that WG profile.
  * iOS: use “On-demand” profiles only if you’re comfortable; otherwise manually toggle but avoid using public Wi-Fi without the VPN.
  * Windows/macOS/Linux: consider firewall rules to block LAN/WAN except via interface `wg-abhaile` or only allow LAN for specific ranges.

### Client behavior per VLAN reach:

* **Admin profile:** full tunnel (`0.0.0.0/0`) + VLAN 20. WAN goes via home WAN. 
* **User profile:** split tunnel – only VLAN 20. Rest goes out local WAN.
* **Travel profile:** split to VLAN 20, 70, 100 (services, cast, DMZ).

## 7. Post–power-outage bootstrap sequence

Assuming full blackout, then power restored.

### Step 1 – Network comes back

* ER605, switches, APs boot and restore VLANs/DHCP/ACLs. 
* Verify from a laptop on VLAN 30:

  * `ping 172.20.20.1`
  * `ipconfig / ifconfig` shows correct subnet, DNS = 172.20.20.235/236.

### Step 2 – Hosts boot with systemd

On **phobos** and **deimos**:

1. `network-online.target` reached – VLAN20/100 IPs + `/32`s configured.

   * `networkctl status` to confirm.
2. Chrony starts, syncs with upstream + peer.

   * `chronyc tracking` on both hosts.

### Step 3 – DNS + NTP ready

* CoreDNS (filtered + clean) start:

  * `systemctl status coredns-filtered coredns-clean`
* Test from admin device:

  * `dig vault.abhaile.home.arpa @172.20.20.235`
  * `dig -x 172.20.20.200 @172.20.20.235`

### Step 4 – Vault startup and auto-unseal

On phobos:

* `vault.service` starts but sealed initially.
* `abhaile-vault-unseal.service` runs:

  * decrypts SOPS file
  * submits unseal keys
* Confirm:

  * `systemctl status vault abhaile-vault-unseal`
  * `vault status` → `Sealed: false`

### Step 5 – Secrets + apps

* Vault Agent renders templates (Caddy, Authelia, SMTP, deSEC, etc.).
* `vault-ready.service` (simple target) can be used:

```ini
[Unit]
Description=Vault is unsealed
Requires=vault.service abhaile-vault-unseal.service
After=vault.service abhaile-vault-unseal.service

[Install]
WantedBy=multi-user.target
```

* Authelia, DMZ Caddy, and SMTP relay have:

```ini
After=vault-ready.service
Requires=vault-ready.service
```

* Internal Caddy starts after Authelia (so SSO backend is ready).

### Step 6 – Application services

* Quadlet services (NetBox, Omada, Home Assistant, etc.) start with dependencies:

```ini
After=caddy-internal.service
Wants=caddy-internal.service
```

* Media stack on deimos starts, but qBittorrent is bound to Gluetun via:

```ini
[Unit]
Description=qBittorrent (via Gluetun)
Requires=gluetun.service
After=gluetun.service
```

### Step 7 – Monitoring & protection

* Prometheus, Alertmanager, Grafana, Loki, exporters come up.
* CrowdSec and Fail2ban services start.
* Validate:

  * Grafana dashboards loading
  * Uptime Kuma shows all key endpoints up.

## 9. Podman quadlets (.volume, .image, .build, .container)

Below are example quadlets for key services. Place them under `/etc/containers/systemd/` (rootful). 

### 9.1 Internal Caddy

**Image descriptor** – `caddy-internal.image`:

```ini
[Image]
Image=docker.io/library/caddy:2
```

**Volume** – `caddy-internal-config.volume`:

```ini
[Volume]
VolumeName=caddy-internal-config
HostPath=/srv/caddy/internal
```

**Container** – `caddy-internal.container`:

```ini
[Unit]
Description=Caddy (Internal)
After=network-online.target vault-ready.service
Requires=vault-ready.service
Wants=network-online.target

[Container]
Image=caddy-internal.image
Network=ipvlan0
PublishPort=172.20.20.200:80:80
PublishPort=172.20.20.200:443:443
EnvironmentFile=/var/lib/abhaile/rendered/caddy/internal.env
Volume=caddy-internal-config.volume:/etc/caddy
Volume=/var/lib/caddy:/data
Exec=sh -c "caddy validate --config /etc/caddy/Caddyfile && caddy run --config /etc/caddy/Caddyfile --adapter caddyfile"

[Service]
Restart=on-failure
RestartSec=5

[Install]
WantedBy=multi-user.target
```

## 10. How to apply these configurations step-by-step

### Step A – systemd units

1. Place unit files under `/etc/systemd/system/`.

2. Validate:

   ```bash
   systemd-analyze verify /etc/systemd/system/*.service
   ```

3. Reload and enable:

   ```bash
   systemctl daemon-reload
   systemctl enable --now chrony coredns-filtered vault caddy-internal
   ```

4. Inspect dependencies:

   ```bash
   systemctl list-dependencies caddy-internal.service
   ```

### Step B – quadlets

1. Place `.container`, `.volume`, `.image`, `.build` files in `/etc/containers/systemd/`.

2. `systemctl daemon-reload`.

3. For each container:

   ```bash
   systemctl enable --now caddy-internal.service
   systemctl status caddy-internal.service
   ```

4. To rebuild images from `.build` quadlets (when used):

   ```bash
   systemctl start custom-app-build.service
   ```

### Step C – CoreDNS & Caddy

1. CoreDNS:

   ```bash
   coredns -conf /etc/coredns/Corefile.filtered -dns.port=1053
   # quick foreground test, then:
   systemctl enable --now coredns-filtered
   ```

2. Caddy:

   ```bash
   podman run --rm -v /srv/caddy/internal:/etc/caddy caddy:2 caddy validate --config /etc/caddy/Caddyfile
   systemctl enable --now caddy-internal.service
   ```

### Step D – WireGuard clients

* Copy the `wg-*.conf` to appropriate clients.
* Scan QR via mobile WG app (`qrencode -t ansiutf8 < wg-admin.conf` on Linux).
